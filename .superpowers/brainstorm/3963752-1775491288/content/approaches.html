Live Transcription Map — Architecture Approaches

How should location detection and map rendering be orchestrated?

A

Backend-Driven (Recommended)

How it works: The existing call transcript WebSocket streams text to the backend. A new LocationExtractor service runs on each transcript chunk — regex for addresses/streets, city name dictionary lookup, and area code mapping. When a location is found, a location_detected event is pushed over the same WebSocket with coordinates.

Geocoding: Nominatim (free, OpenStreetMap-based) or a local city/zip lookup table for common mentions like "Spring Hill", "Franklin", "Mt. Pleasant".

Pros

  • Transcript + location parsing in one place
  • Can match against CRM customer DB server-side
  • No extra API keys or client-side processing
  • Drive time calculation on server (haversine already exists)

Cons

  • Slight latency (~200-500ms) for geocoding round-trip
  • More backend code to maintain
B

Frontend-Driven

How it works: The frontend receives raw transcript text (already happening) and runs a client-side location parser in the browser. When it detects a city/address, it geocodes via a free API (Nominatim) and drops the pin directly. Customer lookup happens via a separate API call.

Geocoding: Direct browser fetch to Nominatim or MapTiler geocoding API.

Pros

  • No backend changes needed for parsing
  • Instant — no round-trip to server for detection
  • Simpler to prototype quickly

Cons

  • Can't easily match against customer DB from browser
  • Nominatim rate limits (1 req/sec) could bottleneck
  • Parsing logic duplicated if other features need it later
  • Heavier browser load during active calls
C

Hybrid — Smart Frontend + Backend Lookup

How it works: Frontend does lightweight keyword detection (city names from a static list, basic address regex). When a candidate location is found, it sends it to a new backend endpoint that geocodes, checks the customer DB, calculates drive time, and determines service zone — returning enriched location data.

Pros

  • Fast initial detection (client-side keyword match)
  • Rich data from backend (customer match, drive time, zone)
  • Best of both approaches

Cons

  • Two codepaths to maintain (frontend parser + backend enrichment)
  • Slightly more complex architecture

💡 My Recommendation: Option A (Backend-Driven)

Your call transcript WebSocket already streams text server-side through Google Cloud Speech. Adding location extraction there means one pipeline handles transcription → location detection → customer matching → zone calculation → drive time — all pushed to the frontend as a single enriched event. The frontend just renders what it's told. Simpler, more powerful, and keeps the map UI thin.
